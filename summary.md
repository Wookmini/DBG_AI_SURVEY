# DB Global Chip AI Tool - 아키텍처 및 보안 검토 요약

**최근 업데이트 일시:** 2026-06-12
**주제:** GitHub Pages + Google Apps Script(GAS) 배포 구조의 보안성 및 대안 검토

---

## 1. 초기 기획 아키텍처
* **프론트엔드:** GitHub Pages (현재 배포 완료 상태)
* **백엔드 (API & DB):** Google Apps Script + Google Sheets
* **특징:** 비용이 발생하지 않으며 서버 관리 불필요. 데이터가 스프레드시트에 곧바로 쌓여 실무자가 확인하고 가공하기 매우 쉬움. 
* **결론:** 기술적으로 매우 훌륭하고 구현 가능한 방식.

## 2. 사내 보안 리스크 (IT 부서 우려 사항)
현재 구조에서 **'사번'과 '성명'이라는 사내 개인정보(PII)**를 수집할 경우, 다음과 같은 치명적 보안 리스크가 발생함:
1. **API 엔드포인트 노출:** 프론트엔드 JavaScript 소스 코드에 GAS 제출 URL이 하드코딩되어, 악성 봇이나 외부인이 URL을 탈취해 가짜 데이터를 쏟아부을(Spamming) 위험.
2. **접근 통제(SSO) 부재:** GitHub Pages는 인터넷에 공개되어 있어, 사외 인원의 접속을 완벽하게 차단할 수 없음.
3. **외부망 데이터 전송:** 사내 개인정보가 퍼블릭망(GitHub)을 거쳐 외부 클라우드(Google)로 흘러가는 구조이므로 전형적인 비인가 IT 시스템(Shadow IT)으로 분류됨.

## 3. 환경 제약 및 대안 한계
* **최고의 대안 (GAS HtmlService):** GAS 자체 기능으로 배포하여 회사 구글 계정 로그인을 강제하는 방안 제안.
  * *제약사항:* **아직 모든 직원이 구글 워크스페이스 계정을 받지 못한 상태**이므로, 계정이 없는 직원이 아예 접속하지 못하는 문제가 발생하여 기각됨.
* **현실적 타협안:** GitHub Pages에 '사내 공통 암호'를 걸고, 구글 시트 백엔드에서 '임직원 명부 교차 검증'을 수행하는 이중 방어막 제안.
  * *제약사항:* 하지만 IT 보안 부서에서 **"개인정보가 외부망으로 나가는 것 자체"를 근본적 리스크로 판단**함에 따라 적용 불투명.

## 4. 최종 해결 및 우회 전략 제안
기획 의도(현업의 자동화 아이디어 취합)를 살리면서 IT/보안 부서의 우려를 원천 차단하기 위한 3가지 전략 도출:

1. **[1안] 개인정보 전면 제거 (가장 권장):** 사번과 성명 수집을 완전히 삭제하고 '부서명(드롭다운)'만 선택하게 하여 **익명 데이터**로 전환. 보안 이슈가 완전히 사라지며 즉시 배포 가능.
2. **[2안] 서버 통신 제거 (이메일 연동):** 구글 시트(GAS)로 데이터를 쏘지 않고, '제출' 클릭 시 사용자의 사내 이메일 프로그램(Outlook)이 열리며 담당자에게 메일을 쏘도록 유도 (`mailto:` 활용).
3. **[3안] 사내망 공식 호스팅:** GitHub Pages를 포기하고, IT 부서에 정식으로 요청하여 만들어둔 HTML/JS 파일들을 사내 인트라넷 서버(사내망)에 배포.

## 5. [업데이트: 2026-06-12] 보안 이슈 해결 및 GAS 연동 구현
* **결정 사항:** 보안 우려가 해소되어 최초 기획했던 **'GitHub Pages (프론트엔드) + Google Apps Script (백엔드)'** 방식을 그대로 확정 및 구현함.
* **수집 데이터:** 타임스탬프, 사번, 성명, 총점, 최종 페르소나(레벨)
* **주요 변경점 (script.js):** 
  * 사용자가 사번/성명 입력 시 전역 변수에 임시 저장.
  * 최종 결과 도출 전 `fetch` API(no-cors 모드)를 사용하여 백그라운드 비동기 통신으로 GAS 웹 앱에 데이터 `POST` 전송.
* **GAS 배포 주의사항 (트러블슈팅):** 
  * 사내 구글 워크스페이스(`dbglobalchip.com`) 환경이므로, 외부망(GitHub)에서 CORS 제약 없이 접근하려면 GAS 배포 시 **"액세스 권한이 있는 사용자: 모든 사용자(Anyone)"** 옵션이 필수적임.
  * 해당 옵션이 제한되어 있을 경우 로그인 리다이렉션(`GET`)이 발생하며 "doGet 함수를 찾을 수 없습니다" 또는 저장 실패 에러가 발생하므로 배포 권한 관리에 유의 필요.

## 6. [업데이트: 2026-06-16] 문항 및 로직 고도화 적용
* **문항 섞기 (랜덤 배치):** 각 문항의 보기가 항상 1번부터 4번 순서로 고정되어 있어 4번만 일괄 선택하는 것을 방지하기 위해 점수(score)는 유지한 채 보기가 화면에 표시되는 순서를 무작위로 섞음 (`Math.random() - 0.5` 활용).
* **흥미 유발형 문항 추가:** 딱딱한 업무 관련 질문 외에 참여자의 흥미를 유도할 수 있도록 일상생활형 문항(배달 음식 선택, 전자기기 구매) 2개를 추가함.
* **점수 체계 조정:** 전체 문항이 6개에서 8개로 늘어남에 따라 각 페르소나별(레벨 1~4) 최소/최대 점수 구간을 재조정함.
* **주관식 설문 개편 (동적 텍스트 입력):** 기존의 3가지 질문 중 'AI를 적용하고 싶은 내 업무'를 삭제하고, 남은 두 질문('현재 쓰고 있는 AI Tool', '경험해 보고싶은 AI Tool')에 대해 사용자가 복수 응답을 쉽게 기재할 수 있도록 **[+] 버튼을 눌러 입력창(행)을 무한히 추가할 수 있는 동적 폼 UI**로 변경함. 콤마를 쓰지 않고 개별 칸에 하나씩 입력하면, 데이터 전송 시 자동으로 `|` 기호로 묶어 하나로 취합 전송됨. 소제목을 "AI에 대한 의견을 들려주세요."로 수정하고 작성을 필수로 강제함.
* **이전/다음 네비게이션 적용:** 보기를 클릭하면 즉시 다음 문항으로 넘어가던 방식을 개선하여, 보기를 선택(하이라이트 표시)한 후 명시적으로 하단의 **[다음]** 버튼을 눌러 넘어가도록 변경함. 또한 이전 문항의 답변을 수정할 수 있도록 **[이전]** 버튼을 추가하였으며, 앞뒤로 이동해도 무작위로 섞였던 보기 순서가 유지되도록 로직을 고도화함. 주관식 설문 화면에서도 이전 문항으로 돌아갈 수 있도록 내비게이션을 확장함.

## 7. [업데이트: 2026-06-17] 안티그래비티(Antigravity) 자동 로그인 관련 문의
* **문의 내용:** 안티그래비티 실행 시 매번 로그인을 요구하는 현상을 해결하고 자동 로그인되도록 설정하는 방법.
* **원인 및 해결안:** 안티그래비티는 기본적으로 인증 토큰을 저장하여 자동 로그인을 지원함. 매번 로그인을 요구한다면 Windows 자격 증명 관리자(Credential Manager) 오류이거나, 사내 보안 정책으로 인해 로컬 인증 토큰이 재부팅 시 삭제되기 때문일 수 있음. 이를 해결하기 위해 Windows 자격 증명 초기화 및 사내 보안 설정 확인을 권장함.

## 8. [업데이트: 2026-06-17] 개인 계정을 활용한 GAS 연동 가이드 A~Z
* **배경:** 사내 보안 정책(DLP 등)으로 인해 사내 계정(dbglobalchip.com) 기반의 GAS Web App 접근 권한을 '모든 사용자(Anyone)'로 개방하기 어려워, 외부 도메인(GitHub Pages)에서의 Fetch POST 요청이 로그인 리디렉션 및 CORS 오류로 차단되는 문제 발생.
* **해결안:** 회사 계정 대신 **새로운 개인 구글 계정(Gmail)**을 생성하여 백엔드 DB용 스프레드시트 및 GAS를 분리 운용.
* **적용 가이드:**
  1. **구글 시트 생성:** 개인 계정으로 스프레드시트 생성 및 첫 행에 타임스탬프, 사번, 이름, 점수, 결과 등급, 사용중인 툴, 도입 희망 툴 헤더 작성.
  2. **Apps Script 로직 작성:** 전송된 JSON 텍스트(`e.postData.contents`)를 파싱하여 스프레드시트에 새 행으로 추가(`appendRow`)하는 `doPost(e)` 함수 구현.
  3. **권한 및 배포:** Web App 배포 시 **"다음 사용자로서 실행: 나"**, **"액세스할 수 있는 사용자: 모든 사용자(Anyone)"**로 권한을 완전히 개방.
  4. **프론트엔드 반영:** 새로 발급받은 URL(`https://script.google.com/macros/s/.../exec`)을 로컬 프로젝트 `script.js`의 `GAS_URL` 변수에 덮어쓰고, 다시 GitHub Pages(`main` 브랜치)로 배포(Push).

## 10. [업데이트: 2026-06-17] Part 2 문항 추가 및 통합 퀴즈 엔진 전면 개편
* **요청 사항:** 기존의 단순 객관식 선택형 질문에서 벗어나, 'Part 1'과 'Part 2'로 나뉘는 다양한 형태(복수 선택, 배타적 선택, 주관식, 동적 추가 입력 등)의 설문 문항(Q1~Q8)을 유연하게 지원하도록 요구됨.
* **주요 개발 내용:**
  1. **다중 선택 및 배타적 선택(Exclusive) 로직 추가:** '아직 활용해 본 업무가 없습니다'와 같은 선택지를 고를 경우 다른 체크박스가 자동으로 해제되고, 반대의 경우에도 해당 선택지가 해제되도록 배타적 처리 로직(`isExclusive: true`)을 구현함.
  2. **통합 주관식(Text) 엔진 도입:** 기존 퀴즈 플로우 밖에서 별도의 페이지(`subjective-view`)로 분리되어 있던 주관식 입력 화면을 없애고, 퀴즈 흐름 내부에서 `type: "text"`(Q4, Q8)를 지원하는 `textarea` 방식의 통합 엔진으로 전면 개편함. 사용자가 빈칸으로 넘기지 못하도록 필수 입력(Validation) 로직도 적용.
  3. **동적 다중 입력(Dynamic Input) 모듈화:** '기타' 항목 선택 시 나타나는 `+` 버튼을 클릭해 입력창의 행(row)을 무한정 추가하거나 `-` 버튼으로 삭제할 수 있는 동적 UI 렌더링 기능을 `options` 설정(`hasDynamicInput: true`)만으로 어느 문항에서나 손쉽게 사용할 수 있도록 확장성 있게 개편함.
  4. **체크박스 UI 개선 (선택 시 시인성 강화):** 문항 다중 선택 시, 누락되어 있던 변수(`--primary-color`) 오류를 수정하고 사용자의 요청에 따라 검은색 톤(테마 기본 텍스트 색상)으로 체크마크(`✔`)가 진하게 그려지도록 CSS(`styles.css`)를 보완함.
  5. **Payload 동적 조립 및 GAS 전송 대비:** Q1부터 Q8까지 늘어난 문항 데이터를 빠짐없이 수집(다중 입력 데이터는 `|`로 구분)하여 구글 시트(GAS)로 한 번에 전송할 수 있도록 `payload` 구조를 최적화함. (이 변경으로 인해 구글 시트의 헤더 및 GAS의 doPost 코드를 Q1~Q8을 받을 수 있도록 사용자가 직접 수정해야 함을 안내함).

## 11. [업데이트: 2026-06-17] 설문 완료 화면 개선 및 데이터 정규화(PDCA 완료)
* **요청 사항:** 마지막 제출 화면 디자인 강화(글래스모피즘), 문구 수정, 그리고 복수 선택(최대 2개) 제약 조건을 기타 항목의 동적 입력(+)에도 동일하게 반영하여 데이터 분석이 쉽도록 보완.
* **주요 개발 내용:**
  1. **완료 화면 UI 고도화:** 딱딱했던 화면을 3D 글래스모피즘 기반의 실버 그라데이션 카드로 변경하고 애니메이션(pulse) 잔상 버그를 해결하여 퀄리티 향상.
  2. **선택 제한 로직 정밀화:** '기타' 항목 내부에 동적으로 추가한 입력창(+)의 개수까지 모두 합산하여 최대 선택 개수(maxSelect: 2) 초과를 정밀하게 차단하도록 로직 개선.
  3. **데이터 분리(정규화) 및 Payload 최적화:** 복수 선택 문항(Q2, Q3, Q5)에서 여러 개의 동적 입력값이 하나의 텍스트로 합쳐지는 문제를 해결. 평탄화(Flattening)를 통해 구글 시트에서 1순위, 2순위 열로 각각 깔끔하게 쪼개지도록 데이터 구조 개편.
  4. **Q1 척도 변환:** Q1(사용 빈도) 항목의 경우, 통계 처리가 용이하도록 긴 텍스트 대신 1~4점 점수(Score)로 치환되어 전송되도록 매핑 로직 추가.
